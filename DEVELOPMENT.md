# Developing Atrium

This covers building Atrium from source and working on it locally. For just
running it, see the [README](README.md).

## Prerequisites

- Docker and Docker Compose
- Node.js 20+ (only needed for frontend hot-reload dev mode, not for building via Docker)

## Setup

```sh
git clone https://github.com/agent-squirrel/Atrium.git
cd Atrium
cp backend/.env.example backend/.env
# edit backend/.env - fill in SECRET_KEY, JWT_SECRET_KEY, POSTGRES_PASSWORD, ENCRYPTION_KEY
```

## Build and run from source

```sh
docker compose up -d --build
```

This builds the backend and web images from the local `backend/Dockerfile`
and `nginx/Dockerfile` (which also builds the frontend) instead of pulling
published packages, and bind-mounts `backend/` into the container so most
backend changes are picked up without a rebuild. Open `http://localhost/admin/`.

## Database migrations

```sh
docker compose exec backend flask db migrate -m "describe the change"
docker compose exec backend flask db upgrade
```

Review autogenerated migrations before committing - Alembic doesn't always
get column type changes or renames right on its own.

## Backend tests

The runtime image only installs `requirements.txt` (production deps); tests
need `requirements-dev.txt` as well:

```sh
docker compose exec backend pip install -r requirements-dev.txt
docker compose exec backend pytest --cov=app --cov-report=term-missing
```

This is also what `.github/workflows/backend-tests.yml` runs in CI on every
push/PR touching `backend/`.

## Frontend hot-reload dev mode

With the Docker stack already running (`docker compose up -d --build`, so
nginx/backend are live on `:80`), run the Vite dev server separately for
instant reload on frontend changes:

```sh
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173/admin/`. `vite.config.ts` proxies `/api`, `/p`,
`/static`, and `/guest` requests through to the Docker stack on `:80`, so
this works without touching CORS on the Flask app.

## Releasing a new version

Versioning is manual semver via git tags - there's no auto-increment. Pushing
a tag matching `vX.Y.Z` triggers `.github/workflows/publish.yml`, which
builds and pushes both images to GitHub Container Registry
(`ghcr.io/agent-squirrel/atrium-backend` and `atrium-web`), tagged with both
the version and `latest`. The version is what shows up at the bottom of
Settings > General in the admin panel.

```sh
git tag v0.2.0
git push origin v0.2.0
```

Bump the version when you ship a schema-changing migration, even though
backup/restore compatibility itself doesn't depend on this number - it
checks the exact Alembic schema revision directly (see
`backend/app/backup.py`), so restores fail safely on a mismatch regardless
of what the version number says. The version is just there to make that
easier to diagnose at a glance.

The first time you publish, go to each package's GitHub page
(`ghcr.io/agent-squirrel/atrium-backend` and `atrium-web`) and set its
visibility to Public - `GITHUB_TOKEN` can publish packages but can't make
them public on its own.
